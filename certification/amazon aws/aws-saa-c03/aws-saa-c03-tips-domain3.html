Core tasks:

  • Right-size compute/storage/DB for the workload profile.
  • Reduce latency via caching and edge (CloudFront/GA), and offload heavy work.
  • Scale horizontally and process in parallel when demand spikes.

Patterns that score

  • Scale-out with ASG/containers/serverless over vertical scaling.
  • Cache at every layer: CloudFront, API caching, ElastiCache (Redis/Memcached).
  • Right storage: EBS gp3/io2, EFS for shared POSIX, S3 for massive throughput.
  • Parallelize with SQS, Kinesis, Step Functions, Lambda concurrency.
  • Accelerate network: Global Accelerator, placement groups, ENA.

Common pitfalls

  • Vertical scaling instead of horizontal.
  • Under-provisioned EBS IOPS/throughput for the DB.
  • No CDN for global users; serving everything from one Region.
  • Chatty synchronous microservices without caching/batching.
  • Forcing relational DBs on key-value/ultra-write use cases (use DynamoDB/Kinesis).

Study checklist

  • EC2 families & Graviton; Auto Scaling policies.
  • EBS types; EFS vs FSx; S3 performance (multi-part, prefixes).
  • RDS vs Aurora vs DynamoDB; read replicas; partitions; WCU/RCU.
  • CloudFront behaviors; Global Accelerator vs Route 53.
  • SQS/SNS/Kinesis/MSK; Lambda concurrency; DLQs.

Exam Focus

  • Pick managed/serverless for perf with less ops.
  • Edge + caching for reads; batch/stream for writes.
  • Match data model to access patterns.
  • Balance performance and cost (Graviton, gp3, S3 Intelligent-Tiering).

Key AWS services

  • EC2 + ASG, ECS/EKS, Lambda
  • ElastiCache, CloudFront, Global Accelerator
  • S3, EBS, EFS, FSx
  • RDS/Aurora, DynamoDB, OpenSearch
  • SQS, SNS, Kinesis, Step Functions, EMR

Hands-on ideas

  • Benchmark gp3 vs io2 EBS.
  • Aurora + read replicas + ElastiCache for read-heavy workload.
  • CloudFront in front of S3 static site and compare TTFB.
