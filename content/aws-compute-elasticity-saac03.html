1. ¿Qué es la elasticidad?

La elasticidad es la capacidad de una arquitectura para **ajustar automáticamente** sus recursos en función de la demanda, manteniendo el SLA y optimizando el coste.

En AWS se logra combinando **Auto Scaling**, **Elastic Load Balancing (ALB/NLB/GWLB)** y **modelos de compra** (On‑Demand/RI/Savings/Spot).

EXAM TIP: Tráfico impredecible → ASG target tracking + ALB + mezcla On‑Demand/Spot.

2. ASG + Load Balancing

Usa **Launch Templates**, **Mixed Instances Policy** y **health checks** de ELB. Activa **cross‑zone** en ALB; considera **NLB** para L4 crítico.

Los **warm pools** reducen la latencia de arranque en picos.

CasoBalanceador recomendado
HTTP con reglas L7ALB
TCP crítico o IPs fijasNLB
Firewall/IDS partnersGWLB

3. Modelos de compra

**Savings Plans** para base estable con flexibilidad; **Spot** para trabajo tolerante a interrupciones con checkpoints; **RI estándar** solo cuando la carga es fija y a largo plazo.

RequisitoOpción
Máximo descuento con flexibilidadSavings Plans Compute
Trabajo batch tolerante a fallosSpot con capacidad diversificada
Base estable 24x7RI/Savings plano 1 o 3 años

4. Políticas de escalado

Target Tracking mantiene una métrica (p.ej., 50% de CPU). Step/Programmatic para escenarios específicos. Programa ventanas con **scheduled scaling**.

Combina **ASG cooldown** + **ALB request count** para picos por tráfico HTTP.

5. Blue/Green & canary

Con **CodeDeploy** + ALB puedes hacer canary/linear y rollback instantáneo via cambio de target group.

EXAM TIP: Piden rollback inmediato → Blue/Green con shift de tráfico.

6. Preguntas tipo examen

Una app web con picos impredecibles necesita costo óptimo y alta disponibilidad. → ALB + ASG target tracking + mezcla On‑Demand/Spot.

Tráfico TCP de baja latencia con IPs fijas. → NLB con cross‑zone y health checks.
