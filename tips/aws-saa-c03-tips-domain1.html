Patrones que puntúan

  • Diseña con IAM mínimo necesario: políticas gestionadas por cliente, condiciones (aws:MultiFactorAuthPresent), roles en lugar de usuarios de acceso programático.
  • Aisla por cuenta/OU con AWS Organizations y SCP para límites preventivos (por ejemplo, DenyChangeLogArchive). Control Tower acelera el baseline.
  • Red por capas: VPC por dominio, subredes privadas por AZ, NACL restrictivas, SG por función. Exposición controlada con ALB/NLB y AWS WAF.
  • Protege datos: S3 con Block Public Access, cifrado S3/KMS, políticas de bucket con condición aws:PrincipalOrgID, EBS/FSx cifrados por defecto, RDS con encryption at rest.
  • Gestión de secretos con Secrets Manager/Parameter Store; rotación automática y acceso mediante IAM o referencias a ARNs.
  • Observabilidad: CloudTrail (org), CloudWatch Logs/Metrics/Alarms, Config con reglas gestionadas y Conformance Packs.
  • Acceso híbrido seguro: VPN/Direct Connect con route filtering y segmentation (TGW + route domains); S2S VPN redundante.

Errores comunes

  • Permitir * en acciones o recursos de IAM.
  • Buckets S3 públicos por políticas heredadas o ACL antiguas. Usa Block Public Access.
  • Falta de rotación de claves KMS/secretos o uso de la clave aws-managed sin control de permisos.
  • Colocar bases de datos en subredes públicas.
  • No registrar CloudTrail en todas las regiones/cuentas.

Lista de estudio

  • Diseño de control de acceso: IAM, SCP, permission boundaries, roles asumiendo desde otras cuentas.
  • Endurecimiento de S3, EBS, RDS, EFS/FSx y cifrado con KMS (CMK vs AWS-managed).
  • Arquitectura de red segura: VPC endpoints, PrivateLink, NAT Gateway, WAF/Shield, Network Firewall.
  • Registro y cumplimiento: CloudTrail org, Config, Security Hub, Detective, GuardDuty.

Pista de examen

Cuando la pregunta hable de múltiples cuentas y controles preventivos, piensa en Organizations + SCP. Para privacidad de datos, prioriza VPC endpoints/PrivateLink y KMS.
